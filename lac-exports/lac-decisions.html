Block Chance System

feat-2026-041
combat frozen 3 decisions

1 Block resolves after the miss check — you can only block attacks that land

Rationale

Keeps the two systems independent and readable. Miss = attacker failed. Block = defender succeeded. Sequential resolution avoids double-negation confusion in log.

Alternatives considered
  • Block rolls before miss check
  • Combined evasion+block single roll
Date2026-03-28

2 Passive 10% base block when any offhand is equipped, even without blockChance stat

Rationale

Makes the offhand slot feel defensively meaningful even for non-block builds. Holding anything in your off-hand is slightly protective.

Alternatives considered
  • Block only from explicit blockChance stat
  • Offhand grants 0% base, all block from affix
Date2026-03-28

3 Block cap at 75% — elemental damage unblockable

Rationale

75% physical block cap matches D2. Elemental bypass distinguishes enchanted monster threat profile and ensures resist gear remains relevant even for shield builds.

Alternatives considered
  • 100% cap
  • Block applies to elemental at 50% effectiveness

Boss Encounters

feat-2026-045
game/combat frozen 3 decisions

1 Boss spawns every 5 floors, not randomly

Rationale

Predictable milestones let players prepare (use town portal, equip best gear). Random boss floors would feel punishing with no agency.

Alternatives considered
  • Random boss among floors 3-7, 8-12 etc — less predictable, less satisfying
Date2026-03-29

2 Enrage triggers at round 5, not a HP threshold

Rationale

Round-based enrage punishes turtling (using potions indefinitely) without being unfair to low-damage builds. HP threshold enrage is invisible and frustrating.

Alternatives considered
  • 50% HP enrage — invisible to player until it happens
  • No enrage — boss is just a big HP bar
Date2026-03-29

3 Cannot flee from boss encounters

Rationale

Boss is a commitment. Fleeing would trivialize the milestone. Players who aren't ready should use a town portal scroll before entering the boss room.

Alternatives considered
  • Allow flee with heavy penalty — dilutes the stakes

Code Quality Session 1 — Timeout Leaks, Render Optimizations & Import Cleanup

feat-2026-059
game/maintenance frozen 4 decisions

1 Both the outer and inner timeouts are tracked by separate let refs (t1, t2) so cleanup always runs clearTimeout on both

Rationale

clearTimeout(undefined) is a safe no-op in JavaScript, so assigning t2 via let before the outer fires is safe — cleanup never throws even if t1 fires and t2 is never assigned.

Date2026-04-01

2 bossWarnTimer cleanup returns clearTimeout on every gameFloor change, not just on unmount

Rationale

When gameFloor changes, the previous floor's boss warning timer should be cancelled — the warning is floor-specific. React runs cleanup before the next effect, which is exactly the right timing.

Date2026-04-01

3 equipped from useInventoryStore is a stable reference for useMemo — bag ops do not change the equipped reference

Rationale

inventoryStore addItem/dropItem only set({ bag, magicFind }) — equipped is not spread. Only equipItem/unequipItem create a new equipped object. So the useMemo for buildPlayerStats invalidates only on real equipment changes, not on every bag mutation.

Date2026-04-01

4 applyZoomWithHaptic not memoized despite being captured in PanResponder closure

Rationale

PanResponder is created once via useRef. applyZoomWithHaptic calls haptics.impactLight() and applyZoom which writes to zoomLevelRef (a ref) — stale closure is not a real problem since the underlying ref is always current. Memoizing would require useCallback and add complexity with no real gain.

Elemental Damage & Resistance System

feat-2026-039
combat frozen 4 decisions

1 Elemental damage is additive on top of physical, never replaces it

Rationale

A fire ring should always add value. If elemental replaced physical, low-physical builds would be punished for stacking resist gear.

Alternatives considered
  • Elemental damage replaces physical for that hit
  • Separate elemental attack action
Date2026-03-28

2 Resist cap at 75%, same as Diablo 2

Rationale

Prevents full immunity builds that trivialize enchanted monsters. 75% is a well-tested ceiling that still makes resist gear feel powerful.

Alternatives considered
  • 100% resist cap (full immunity possible)
  • 50% cap (resist gear less valuable)
Date2026-03-28

3 Status effects last exactly 1 round — chilled, burning, lightning bypass

Rationale

Simple and readable. Multi-round tracking adds complexity without meaningful strategic depth at this game's combat scale (3-6 rounds typical).

Alternatives considered
  • Multi-round DoT stacks
  • Permanent until cured with a potion
Date2026-03-28

4 Block (offhand) only stops physical damage, not elemental

Rationale

Gives enchanted monsters a distinct threat profile — a shield build is not invincible against a Fire Enchanted monster, creating target prioritization.

Alternatives considered
  • Block stops all damage types
  • Separate elemental resist equipment slot

Attack Accuracy & Miss Chance

feat-2026-040
combat frozen 3 decisions

1 Hit chance range clamped 20%-95% for both player and monsters

Rationale

Never feels completely impossible (20% floor) and always has some uncertainty (95% ceiling). Removing all misses or all hits breaks tension.

Alternatives considered
  • Uncapped (0-100% possible)
  • Fixed 85% base with no dexterity modifier
Date2026-03-28

2 Dexterity drives both offensive accuracy and defensive dodge

Rationale

Makes dexterity a dual-purpose stat — relevant for fast/evasive builds AND precision/critical builds. One stat, two payoffs.

Alternatives considered
  • Separate accuracy and dodge stats
  • Dexterity only affects crit chance (current behavior)
Date2026-03-28

3 Monster speed drives evasion — fast monsters are harder to hit

Rationale

speed is already a MonsterDef field and affects turn order. Reusing it for evasion gives it combat meaning without new monster data.

Alternatives considered
  • Separate monster evasion stat in MonsterDef
  • All monsters have same evasion regardless of type

Infinite Difficulty Tiers

feat-2026-038
progression frozen 6 decisions

1 Infinite numeric tiers instead of named Normal/Nightmare/Hell

Rationale

No hard ceiling means the designer can always introduce harder content. Named tiers imply finality; numbers imply an open ladder.

Alternatives considered
  • 3-tier D2 model (Normal/Nightmare/Hell)
  • Prestige system that resets gear on tier unlock
Date2026-03-28

2 Tier persists across deaths; floor resets to 1 on death within a tier

Rationale

Death is a floor-level setback, not a tier-level setback. Losing a tier would feel too punishing given the graveyard sacrifice also occurs.

Alternatives considered
  • Death drops you one tier
  • Death resets both tier and floor
Date2026-03-28

3 Monster multiplier: HP = 1 + (tier-1)*0.5, damage = 1 + (tier-1)*0.35

Rationale

Additive scaling prevents absurd numbers at high tiers while keeping each tier noticeably harder. Tier 5 monsters have 3x HP — challenging but not impossible with tier-appropriate gear.

Alternatives considered
  • Exponential multiplier (doubles each tier)
  • Only damage scales, not HP
Date2026-03-28

4 Loot MF bonus: (tier-1)*20 added to magicFind for all loot rolls

Rationale

Ensures higher tiers are always more rewarding in quality, not just harder. The gear treadmill needs a carrot, not just a stick.

Alternatives considered
  • Flat loot table per tier (new item pool)
  • No loot quality boost — only difficulty increases
Date2026-03-28

5 Track first-time tier clears in careerStats.tiersCleared: number[]

Rationale

Allows the TierClearOverlay to distinguish a first-time clear from a repeat. First clears get a special '✦ FIRST CLEAR ✦' eyebrow for recognition without permanently changing the flow.

Alternatives considered
  • Boolean flag per tier in a map
  • Always show first-clear banner (no tracking)
Date2026-03-31

6 Ascension mechanic: ASCEND button on victory screen pre-loads Ghost Echo from last graveyard entry

Rationale

Ascension is a deliberate prestige-style restart that keeps tier progress, increments an ascension counter, and rewards the player with a Ghost Echo head-start — making the loop feel rewarding rather than punitive.

Alternatives considered
  • Ascension just restarts with no bonus
  • Ascension drops one tier

Room Grammar — Charnel & Sanctum Encounter Bias

feat-2026-055
game/combat frozen 2 decisions

1 Classify rooms by geometry (aspect ratio + area) rather than random assignment

Rationale

Seeded generation means room shapes are deterministic — classifying by geometry makes the grammar consistent and learnable. A Charnel room is always narrow; the player's visual read of the corridor predicts the encounter type.

Alternatives considered
  • Random per-room type tag — breaks the visual grammar and makes the dungeon unreadable
  • Hand-placed special rooms — requires non-BSP generation and loses the procedural seed guarantee
Date2026-04-01

2 Bias encounter weights, not override them — room type adjusts probabilities, not outcomes

Rationale

A Charnel room that guarantees an elite every step would feel oppressive and exploit-prone. Biasing weights preserves variance and the seeded PRNG's run-reproducibility while still making the room type meaningful.

Alternatives considered
  • Hard override (Charnel always spawns elite) — predictable to the point of feeling scripted

Run Meta Layer — Heatmap, Echo Whisper & Graveyard Study

feat-2026-057
progression frozen 3 decisions

1 Graveyard Study is irreversible — studying an entry permanently marks it and prevents echo invocation

Rationale

The value of the study mechanic comes from the decision: echo (temporary floor power) vs. knowledge (permanent record). If study were reversible the choice would have no weight. The sacrifice framing mirrors the graveyard's existing death-penalty philosophy.

Alternatives considered
  • Study as a copy (keep entry + gain bonus) — removes the tension entirely
Date2026-04-01

2 Study bonuses are flavor-only in v1 — descriptions shown in Graveyard, no mechanical effect

Rationale

Mechanical effects (e.g. extended fog radius, auto-identification) require changes across multiple engine files and need balance testing. Shipping flavor bonuses first validates the core decision mechanic without risking balance. Effects can be layered in v2 once the careerStats.studiedBonuses[] array is populated with real player data.

Alternatives considered
  • Ship 1-2 mechanical bonuses with flavor — increases scope and bug surface in a pre-App Store window
Date2026-04-01

3 Echo Whisper fires once per run, on floor 1 only, with a 2.5s delay

Rationale

Firing on floor 1 gives the whisper context — the player just invoked the echo and is about to explore. The 2.5s delay ensures the player has settled into the grid view before the text appears. Once-per-run prevents the whisper from becoming intrusive noise across floors.

Alternatives considered
  • Fire on every floor with echo active — too frequent, loses the 'mentor speaks rarely' atmosphere
  • Fire immediately on mount — competes with existing mount effects (legend hint, ghost echo hint)

Run Stakes — Pre-Run Intent with Bonus Drop

feat-2026-058
game/progression frozen 3 decisions

1 activeStake persists across runs — not reset by startRun

Rationale

The stake represents the player's declared hunt style, not a per-run commitment. A player who always wants to slay should not have to re-select every run. stakeClaimed resets each run, so the reward mechanic still triggers once per completion.

Date2026-04-01

2 35% bonus drop probability — not guaranteed

Rationale

A guaranteed drop would make stakes feel like a farming mechanic — players would optimise for the fastest completable stake rather than their preferred playstyle. 35% is frequent enough to feel meaningful but rare enough to stay exciting. The RNG is seeded from (seed ^ constant ^ floor) to maintain run reproducibility.

Alternatives considered
  • 100% drop — trivially farmable
  • 10% drop — feels unrewarding, players stop caring about claims
Date2026-04-01

3 Stake bonus item delivered directly to bag — no new loot screen

Rationale

Opening a loot screen mid-run for a single item would break the movement flow and require wiring a new loot entry point. Directly calling inventoryStore.addItem() with a toast notification delivers the reward without interrupting the game loop.

Alternatives considered
  • New 'stake loot' screen — higher friction, requires additional state routing

Persistent Save / Load — AsyncStorage Serialization

feat-2026-050
game/persistence frozen 3 decisions

1 Save only cross-run state — bag is NOT saved

Rationale

The bag is run-scoped by design. Saving it would mean a crashed run persists its loot state, which undermines the roguelite death-resets-run contract. Only stash, equipped, xp, level, graveyard, and class survive.

Alternatives considered
  • Save everything including bag — breaks roguelite death loop
  • Save nothing — already the broken current state
Date2026-03-29

2 Single save slot with version field, no multi-slot in v1

Rationale

Multi-slot save requires a save slot selection UI before gameplay. For a solo mobile game in v1, one save slot is sufficient and keeps hydration path simple.

Alternatives considered
  • 3 save slots — good for future, unnecessary complexity now
Date2026-03-29

3 500ms debounce on persist writes

Rationale

gainXp can be called multiple times per combat (level-up check). Writing to AsyncStorage on every XP tick would cause excessive I/O. 500ms debounce batches a full combat sequence into one write.

Alternatives considered
  • Write on every state change — too frequent
  • Write only on app backgrounding — risk of loss on crash
Date2026-03-29

Shared Stash & Town Screen

feat-2026-044
game/ui frozen 5 decisions

1 Stash is accessible only in town (TownScreen), not from dungeon loot screen

Rationale

Dungeon stashing broke the intended flow: pick up items → manage in town. Keeping stash town-only creates a deliberate decision loop: survive the floor → return to town → deposit keepers → start next run.

Alternatives considered
  • Stash accessible anywhere — removes the town trip incentive
  • Stash accessible from inventory screen (in-run) — still breaks the intent
Date2026-03-29

2 Visiting town from the dungeon requires a Town Portal Scroll (rare consumable) — no scroll, no town

Rationale

A potion cost was too cheap — players have 4 potions per run and would visit town freely. Scrolls are gated by drop luck: you only get them from chests (always) and elite/rare kills (10-20% chance). This makes each town visit a deliberate, rare event rather than a habit. Inspired directly by Diablo 2's Town Portal scroll mechanic.

Alternatives considered
  • Potion cost — too available, doesn't create real scarcity
  • HP cost (25% max) — always possible, doesn't reward smart play
  • Town only accessible between floors — too restrictive, blocks emergency use
Date2026-03-29

3 TownScreen combines character sheet + stash management in a tabbed UI

Rationale

Players need both views in sequence: see current stats, see what's in stash, decide what to equip. A combined screen reduces navigation friction. Two tabs (EQUIPPED / STASH) keep it scannable.

Alternatives considered
  • Separate CharacterScreen and StashScreen — extra navigation step
  • Stash as a modal overlay on the main menu
Date2026-03-29

4 Equipping from stash displaces the current item back to stash (not to bag)

Rationale

Bag space is limited (20 slots). Stash is the intended storage layer. Sending the displaced item to the bag would fill it up during normal stash management. Stash-to-stash swaps keep the bag free for dungeon pickups.

Alternatives considered
  • Displaced item goes to bag — can overflow bag unintentionally
  • Must manually unequip before equipping a new stash item
Date2026-03-29

5 Shared stash is a flat Item[] in gameStore (Zustand), not a separate store or SQLite

Rationale

The stash is logically global game state (like graveyard, xp, tier). Zustand keeps all persistent state co-located. SQLite is reserved for future full save-slot implementation. A flat array is sufficient for v1 — no size cap enforced.

Alternatives considered
  • inventoryStore: add stash section — confusing with run-scoped bag
  • Separate stashStore — unnecessary split for co-located persistent state
Date2026-03-29

D2Game — iOS Grid RPG

feat-2026-034
game/mobile frozen 5 decisions

1 Seeded PRNG (mulberry32) for all randomness

Rationale

Every run must be reproducible from a seed. This enables deterministic bun tests, bug reproduction via seed+steps, and future daily-challenge mode. Non-seeded Math.random() would make tests flaky.

Alternatives considered
  • Math.random() — not seedable, untestable
  • xoshiro128 — slightly better distribution but mulberry32 is simpler and sufficient
Date2026-03-28

2 Engine-first development: all game logic written and bun-tested before any UI screen exists

Rationale

Game systems (loot, combat, grid) are pure data transformations. Keeping them React-free means they can be tested headless, simulated at scale (10k items, 1000 combats), and iterated without a device. UI that depends on untested engine code leads to hard-to-debug emergent bugs.

Alternatives considered
  • Build UI first, engine emerges from component state — common in game jams, hard to test
  • Unity/Godot — built-in test frameworks but abandons RN expertise
Date2026-03-28

3 Combat as a fast gate (3–6 rounds, numbers-only UI)

Rationale

Playtesting Diablo 2 and similar games shows the dopamine comes from the drop, not the fight. Combat complexity competes for design budget with loot and inventory. Keeping combat to a transaction screen (<5s) lets players do more runs and see more loot.

Alternatives considered
  • Turn-based tactics with positioning — fun but requires 10× more design work
  • Real-time action combat — incompatible with a grid-step model
Date2026-03-28

4 Normal (white) items are valuable via the socket/runeword system

Rationale

D2’s most memorable items are runewords built into white bases. If white items are junk, players stop picking them up and loot excitement drops by 50%. Every white drop with sockets must show a tooltip flag: ‘Runeword candidate (N sockets)’.

Alternatives considered
  • Only magic+ items are interesting — simpler but loses D2 DNA
  • Crafting system — too complex for v1
Date2026-03-28

5 10×4 Tetris inventory grid with physical item sizes

Rationale

Inventory management as a spatial puzzle is one of D2’s most beloved secondary systems. Charms provide passive bonuses but occupy space, creating a real tradeoff that drives decisions mid-run. A flat list inventory eliminates this tension.

Alternatives considered
  • Flat list with item count limit — simpler but no spatial tension
  • Unlimited stash — removes one of the core decision loops
